## Decisions made

- **Loop discipline over completeness.** Rather than over-engineering one milestone, the loop shipped M10–M14 in sequence with the same plan → review → merge → push rhythm. Each milestone got code-lead review and HOLD fixes when warranted (M11 had 1 BLOCKING + 1 Critical + 6 Major; M12 had 3 Major; M13 had 1 Major on its second pass). Speed came from rhythm, not from skipping review.
- **`docs/forecasts/.pending-forecast.md` handoff file (M11)** — the cleanest way to let `/g-plan` hand a not-yet-saved plan to `/g-forecast` without rewriting either skill to share in-memory state. Cleaned up at the end of Step 4 regardless of outcome.
- **Ratio-based telemetry derivation (M12)** — replaced absolute ⚠ count with `warn_ratio = warns / computable` after code-reviewer found that `n/a` metrics on a thin corpus made `recovery` unreachable. Floor at `computable < 3 → stable` subsumes the bootstrapping case cleanly.
- **HOLD counter is unconditional (M12)** — initial design only incremented `.claude/review-holds` under `defensive`/`recovery`, but code-reviewer found the asymmetry biased the rework signal. Made the increment unconditional; only the *consumption* of the profile changes with state. `/g-telemetry` resets on `stable`.
- **Supplementary profile pattern (M13)** — `frontend-data-flow` is the first supplementary profile in the project. Auto-installs alongside any component framework but never replaces the primary architect. Likely template for future cross-cutting profile additions.
- **Cold-start guards everywhere (M11/M12/M14)** — every new intelligence skill (`/g-forecast`, `/g-telemetry`, `/g-identity`) has an explicit thin-corpus refusal path. Fabricating insights from insufficient signal would be worse than no insight.

## Patterns

### Worked well

- **Parallel architect + code-reviewer dispatch** — every milestone after M10 ran both in parallel for the initial review pass, then code-lead with attestation of resolved findings. This stayed under the 7-agent-per-task cap and cut wall-clock significantly vs. serial review.
- **Reviewer caught real bugs** — code-reviewer surfaced legitimate Major findings in three of five milestones (`grep -c` integer parsing in M11, `/g-retro` plan-slug undefined in M11, telemetry `n/a` profile derivation in M12, stale code-lead prompt in M13). Fast review pass → fix → re-review pattern paid off each time.
- **Self-referential design (M11–M14)** — every new intelligence skill that consumes `docs/retros/` reused the M10 sentinel filter. Single rule, four skills follow it. No duplication.
- **Done-conditions ticked before code-lead** — after the M11 finding about unchecked checklists despite ✅ Complete, every subsequent milestone ticked the boxes before review. Reduced one Minor finding per milestone.

### Avoid / do differently

- **Stale list synchronisation in `/g-specialize`** — M13 shipped with three identical alphabetical stack lists (description frontmatter, no-brief prompt, code-lead consultation prompt) that drifted out of sync; code-lead caught two of them stale on first pass. For M15 or future profile additions: either centralise the list in one place and reference it, or add a `/g-doctor` check that grep-greens all three locations for the same set.
- **Sub-batch semantics under-specified in first M12 pass** — `/g-execute` recovery-mode sub-batches W3.1/W3.2 needed an explicit Rule about serial dispatch and BLOCKED-stops-wave. Added in the fix pass but should have been in the initial Step 0 design. For future adaptive-orchestration changes: enumerate the wave-completion gate semantics under every applicable profile *before* the review.
- **Documentation currency check at code-lead time, not earlier** — three of the five milestones had stale README counts, stale ROADMAP handoff blocks, or stale list strings that code-lead found. Faster turn: include a "release pass cross-check" as the last wave of every milestone, not as a downstream code-lead finding.
- **Skill count drift on profile-only milestones** — M13 added zero new skills but changed the marketplace description string; the M13 fix had to update only the agent and profile counts. For future profile-only milestones: explicit reminder that skill count is *unchanged* prevents accidentally bumping it.
